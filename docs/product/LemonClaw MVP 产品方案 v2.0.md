# LemonClaw MVP 产品方案 v2.0

## 1. 产品概述

### 1.1 产品名称

**LemonClaw**

### 1.2 一句话定位

LemonClaw 是一款面向个人高频 AI 用户的桌面 AI Agent 工作台，帮助用户以图形化方式管理和使用基于 OpenClaw 的本地 Agent 系统，并在真实任务处理中逐步沉淀可复用技能。

### 1.3 产品目标

LemonClaw 的目标不是重新发明一套 Agent Runtime，而是在 OpenClaw 既有能力之上，构建一个真正可安装、可配置、可使用、可观察、可逐步复用经验的桌面产品。

它要解决的问题包括：

- OpenClaw 原生能力偏技术用户，日常使用门槛较高
- 单纯的配置面板无法支撑高频使用，缺少真正的工作台入口
- 会话、任务、技能和 Agent 配置之间缺少统一的桌面产品层组织
- 长会话、重复任务和多 Agent 使用场景需要更清晰的产品化承接

---

## 2. 第一阶段核心用户

### 2.1 核心用户定义

LemonClaw 第一阶段以**产品创建者本人及相似类型的个人高频 AI 用户**为核心用户。

这类用户通常具备以下特征：

- 已经在高频使用 ChatGPT、Claude 或 API 型 AI 工具
- 有文件处理、资料整理、调研分析、配置排查、重复任务执行等需求
- 能接受桌面安装，但不希望长期依赖 CLI 和复杂配置文件
- 希望 AI 不只是回答问题，而是真正参与任务执行
- 希望常见任务能够逐步沉淀经验，而不是每次都从零开始

### 2.2 扩展目标用户

在第一阶段稳定后，LemonClaw 可逐步扩展到：

- 个人知识工作者
- 独立开发者
- 内容与运营从业者
- 需要本地 Agent 工作台的中度 AI 用户

## 3. 产品定位与核心价值

### 3.1 产品定位

LemonClaw 的主定位是：

> **桌面 AI Agent 工作台**

它不是：

- 纯聊天机器人
- 单纯的 OpenClaw GUI 配置器
- 企业级多租户平台
- 复杂工作流编排系统
- 多渠道运营平台

它是一个围绕本地 Agent 使用闭环构建的桌面产品，既承接 OpenClaw 的底层执行能力，也承接用户在日常任务中的交互、管理与经验复用。

### 3.2 核心价值

LemonClaw 第一阶段的核心价值聚焦为三点：

#### 易用

通过桌面图形界面替代命令行和复杂配置，降低 OpenClaw 的使用门槛。

#### 可用

让 OpenClaw 的 Gateway、本地运行、模型配置、会话管理和工具能力，真正变成可启动、可连接、可观察、可恢复的产品能力。

#### 可复用

让重复任务逐步沉淀为可复用的工作资产，而不是每次重新开始。  
这里的复用既包括：

- 已保存的任务会话
- 已确认的技能候选
- 可重复使用的 Agent 任务处理方式

### 3.3 产品原则

LemonClaw 在第一阶段遵循以下原则：

- **工作台优先**：先成为一个真正能用起来的桌面 AI Agent 工作台，而不是功能堆叠的集合
- **复用底层**：尽量复用 OpenClaw 既有底层能力，不重写 Runtime 和 Gateway
- **本地优先**：优先围绕本地运行、本地配置、本地会话管理和本地服务入口构建桌面产品体验
- **主线稳定**：先跑通核心使用闭环，再逐步扩展高级能力
- **渐进增强**：复杂规则、深度自动化和更重的治理能力后置
- **会话治理**：将会话视为工作单元，而不是一次性聊天记录
- **能力可见**：让配置、状态、模型、会话、技能尽可能可见、可理解、可管理

## 4. 参考项目与能力来源

### 4.1 整体能力来源

LemonClaw 不是简单拼装多个项目，而是在不同层级借鉴不同能力来源。

| 能力板块                           | 主要来源            | LemonClaw 借鉴方式                                           |
| ---------------------------------- | ------------------- | ------------------------------------------------------------ |
| Gateway / 统一服务入口与会话路由层 | OpenClaw            | 直接复用，作为本地服务入口、消息承接层和会话路由基础         |
| 多渠道消息与会话入口               | OpenClaw            | 承接 OpenClaw 的多渠道消息接入与会话路由能力，但第一阶段以桌面工作台入口为主 |
| Agent Runtime / 执行引擎           | OpenClaw            | 直接复用底层执行能力                                         |
| Session / Compaction 基础能力      | OpenClaw            | 直接复用，并做工作台化封装                                   |
| Tool Calling / MCP                 | OpenClaw            | 直接复用                                                     |
| 桌面封装与安装配置主线             | RivonClaw           | 借鉴产品化路径与本地产品层状态承接思路                       |
| Chat UI / 多会话交互               | ClawX               | 借鉴交互层设计                                               |
| 技能沉淀与复用机制                 | Hermes              | 借鉴任务到技能的闭环机制与技能文档化方式                     |
| 轻量多 Agent 模式                  | OpenClaw + HomiClaw | 轻量化引入                                                   |
| Session 治理思路                   | HomiClaw            | 借鉴长会话治理与结构化压缩思路                               |
| Fallback 策略                      | HomiClaw            | 作为后续增强参考                                             |

### 4.2 各项目在 LemonClaw 中的角色

- **OpenClaw**：底层运行底座，提供以 Gateway 为核心的统一服务入口、多渠道消息接入、Runtime、Session、Tools、MCP 等基础能力
- **RivonClaw**：桌面产品化封装参考，重点在安装、配置、生命周期管理，以及桌面产品层对本地状态与设置的承接方式
- **ClawX**：桌面交互参考，重点在 Chat UI、多会话和主工作台体验
- **Hermes**：经验沉淀参考，重点在任务复盘、技能候选、技能文档化与相似任务复用，不直接照搬其完整记忆插件体系
- **HomiClaw**：局部机制参考，重点在轻量多 Agent 和会话治理，不作为整体架构照搬对象

## 5. 产品形态

### 5.1 产品形态定义

LemonClaw 是一个以桌面为主要载体的 AI Agent 工作台，包含两层体验：

#### 主工作台

- 任务输入
- 对话与文件交互
- 会话切换
- 结果查看
- Agent 使用入口

#### 管理与配置层

- Gateway 管理
- 模型与 API 配置
- Agent 配置
- 技能启用与查看
- 状态与日志查看

### 5.2 产品不追求的形态

第一阶段不把 LemonClaw 做成：

- 企业级 AI 中台
- 通用工作流平台
- 多人协作管理系统
- 深度自动编排中心
- 复杂权限审批系统
- 多渠道运营平台

### 5.3 第一版的目标形态

第一版的目标不是功能面面俱到，而是：

> **先把 LemonClaw 作为桌面 AI Agent 工作台的核心使用闭环跑通。**

也就是说，用户能够：

1. 装起来
2. 连上 OpenClaw
3. 配好模型
4. 在工作台里发起任务
5. 看见结果和会话
6. 让任务经验开始沉淀

### 5.4 MVP 核心使用闭环

LemonClaw 第一版的最小闭环如下：

1. 用户下载安装 LemonClaw
2. 首次启动时系统检测并初始化 OpenClaw 运行环境
3. 用户配置模型 Provider 和 API Key
4. 用户启动或连接 OpenClaw Gateway
5. 用户进入主工作台，新建一个会话
6. 用户选择一个 Agent 并发起任务
7. Agent 返回结果，系统展示会话内容和基础状态
8. 会话被持久化保存，可再次打开并继续
9. 任务结束后可生成任务复盘或技能候选记录

## 6. 能力板块划分

### 6.1 Gateway / 统一服务入口与会话路由层

LemonClaw 依赖 OpenClaw Gateway 作为本地统一服务入口和会话路由层。

这一层负责：

- 本地服务启动与停止
- 请求入口承接
- 会话路由
- 与 Runtime 的衔接
- 多入口消息的统一承接基础
- 本地 Agent 服务承载

LemonClaw 在这一层的主要职责是：

- 图形化管理 Gateway 生命周期
- 检测连接状态
- 向用户暴露必要但不过度复杂的状态信息
- 承接 Gateway 的配置、日志和健康状态展示

### 6.2 Agent Runtime / 执行引擎

Runtime 负责：

- Agent 执行循环
- LLM 调用
- Tool 调用
- 会话上下文参与执行
- 子 Agent 执行基础能力

LemonClaw 不重写 Runtime，而是在产品层组织和调用它。

### 6.3 LLM 管理

LLM 管理是 LemonClaw 的核心能力板块之一，不应仅被视为普通设置项。

这一层包括：

- Provider 管理
- API Key 配置
- 模型选择
- 主模型绑定
- 备用模型 / fallback 预留
- 不同 Agent 的模型绑定
- 模型连通性测试

在 LemonClaw 中，**LLM 管理属于资源层能力**，而 **Agent 配置属于行为层能力**。  
第一阶段需要明确区分模型资源配置与 Agent 角色配置，避免两者混在一起。

### 6.4 Session / 会话管理

Session 在 LemonClaw 中不是简单聊天记录，而是**工作台中的任务会话单元**。

它承载：

- 用户输入与回复历史
- Agent 执行轨迹
- 文件与上下文关联
- 当前任务状态
- 后续恢复能力
- 技能沉淀的来源信息

LemonClaw 将复用 OpenClaw 的 Session / Compaction 基础能力，并参考 HomiClaw 的治理思路，将会话做成可持续管理的工作单元。

### 6.5 Skills / Tools / MCP 管理

这一层承接 OpenClaw 的工具与扩展能力，包括：

- Tool calling
- MCP 接入
- Skills 机制
- 工具能力承载

LemonClaw 第一阶段的重点不是重新设计工具体系，而是把它们以更清晰的方式暴露出来，包括：

- 技能启用 / 禁用
- 可用工具和技能查看
- 工具使用结果可见化

### 6.6 多 Agent 管理

LemonClaw 将引入**轻量多 Agent 模式**，但不引入 HomiClaw 的团队和主管模型。

第一阶段的多 Agent 重点包括：

- Agent 注册与预设
- 会话绑定 Agent
- 多个专长 Agent 可切换使用
- 每个 Agent 具备独立角色、模型绑定、工作空间和技能偏好

更复杂的子 Agent 委派和并行编排作为后续增强方向，而非第一版主线。

### 6.7 桌面工作台 / 交互层

桌面交互层是 LemonClaw 的产品主入口，重点参考 ClawX 和 RivonClaw。

它包括：

- 主聊天工作台
- 会话列表
- 文件输入
- 结果展示
- 状态与日志入口
- 设置与配置入口

### 6.8 技能沉淀与复用

这一层主要借鉴 Hermes 的思路，但做更克制的产品化表达。

LemonClaw 第一阶段不宣传"自我进化"，而强调：

- 任务复盘
- 技能候选记录
- 用户确认保存
- 相似任务逐步复用

### 6.9 多渠道消息能力的产品定位

OpenClaw 的多渠道消息接入能力是其重要差异化优势之一，而 Gateway 是这一能力的核心承载层。  
但对 LemonClaw 而言，第一阶段不以"多渠道平台"作为产品主线，而是优先聚焦桌面工作台闭环。

这意味着：

- LemonClaw 在产品认知上承认并保留 OpenClaw 的多渠道能力基础
- LemonClaw 通过 Gateway 预留未来承接多渠道消息能力的扩展空间
- 第一阶段主要聚焦桌面端任务入口、会话管理和产品层状态承接
- 多渠道接入、渠道配置和跨渠道会话运营能力作为后续扩展方向，而非 MVP 主路径

## 7. 核心场景

### 7.1 场景一：桌面任务工作台

用户在 LemonClaw 中发起真实任务，例如：

- 总结一份文档
- 分析一个项目仓库
- 整理一段资料
- 检查配置文件
- 处理一个本地文件任务

用户在工作台中完成：

- 输入任务
- 上传文件
- 选择 Agent
- 查看执行结果
- 保存会话

### 7.2 场景二：本地 Agent 系统的图形化管理

用户无需频繁回到 CLI，即可完成：

- 启动 / 停止 Gateway
- 配置模型和 API Key
- 检查 Agent 状态
- 管理基础设置
- 查看日志和错误

### 7.3 场景三：多 Agent 的角色分工使用

用户针对不同类型任务使用不同专长 Agent，例如：

- 通用助手
- 配置助手
- 文档助手
- 调研助手

不同会话绑定不同 Agent，减少上下文污染，并提升使用清晰度。

### 7.4 场景四：会话持续与任务恢复

用户在一次任务未完成时可以关闭工作台，之后再回来继续。

LemonClaw 应支持：

- 会话持久化
- 历史恢复
- 基础状态回看
- 长会话的后续压缩治理

### 7.5 场景五：任务经验逐步沉淀

当某类任务具有重复性时，系统可以在任务结束后生成复盘，并进一步形成技能候选记录，供用户保存和后续复用。

### 7.6 场景六：多渠道能力的后续承接

在后续版本中，LemonClaw 可以逐步承接 OpenClaw 的多渠道消息能力，例如：

- 渠道接入配置
- 渠道会话查看
- 不同入口的会话映射
- 基础的渠道状态管理

但这部分不作为第一版的主要交付目标。

## 8. Session 与会话治理

### 8.1 Session 的产品定义

在 LemonClaw 中，Session 不只是聊天记录，而是承载任务目标、执行进展、关键决策和后续待办的工作单元。

### 8.2 Session 基础能力

第一阶段的 Session 应支持：

- 会话创建
- 会话持久化
- 会话恢复
- 会话列表展示
- 标题与更新时间管理
- 与 Agent 的绑定关系

### 8.3 Session 压缩的正确定位

Session 压缩的底层能力来源于 OpenClaw。  
LemonClaw 借鉴 HomiClaw 的不是"是否有压缩"，而是：

- 如何把压缩结果组织为结构化摘要
- 如何保留任务连续推进所需的关键信息
- 如何让长会话仍然可恢复、可继续、可管理

### 8.4 LemonClaw 对压缩能力的理解

LemonClaw 中的压缩不应只是"历史摘要"，而应更接近：

> **任务状态保真压缩**

即在长会话场景下，尽量保留：

- 任务目标
- 关键决策
- 待办事项
- 用户偏好与约束
- 关键路径、URL、ID 等精确标识符
- 最近若干轮原文上下文

### 8.5 Session 压缩与技能沉淀的关系

结构化压缩后的会话摘要，在后续版本中也可作为任务复盘和技能候选提取的补充输入来源之一。

## 9. 多 Agent 方案

### 9.1 多 Agent 的产品定义

LemonClaw 支持面向个人用户的轻量多 Agent 工作模式。

这意味着：

- 用户可在一个工作台中使用多个专长 Agent
- 不同会话可以绑定不同 Agent
- 每个 Agent 有自己的定位、配置和工作空间

### 9.2 第一阶段纳入的多 Agent 能力

第一阶段建议纳入以下能力：

- Agent 预设列表
- 新建会话时选择 Agent
- 会话与 Agent 的绑定关系
- 每个 Agent 独立角色描述
- 每个 Agent 独立模型绑定
- 每个 Agent 独立工作空间
- 每个 Agent 的技能偏好或技能集配置

### 9.3 第一阶段不纳入的多 Agent 能力

第一阶段不纳入：

- 团队主管模型
- teamIds 组织架构
- 多层级企业协作模型
- 复杂 allowAgents 白名单系统
- 并行多子 Agent 编排
- 完整的子 Agent 可视化控制面板
- 子 Agent 自动委派与复杂协作编排

### 9.4 后续演进方向

在后续版本中，LemonClaw 可逐步探索：

- 主 Agent 向专长 Agent 发起简单委派
- 子 Agent 任务回传
- 更细的上下文可见性控制

## 10. 技能沉淀与复用机制

### 10.1 设计定位

LemonClaw 第一阶段不使用"自我进化"作为核心卖点，而使用更务实的表达：

> **技能沉淀与复用机制**

### 10.2 基础闭环

第一阶段的技能沉淀闭环为：

1. 用户完成任务
2. 系统生成任务复盘
3. 系统生成技能候选记录
4. 用户确认是否保存
5. 保存为结构化技能文档
6. 后续任务中逐步复用

### 10.3 机制原则

第一阶段技能机制遵循以下原则：

- 优先服务高频、可复用任务
- 用户可见、可确认、可管理
- 不追求全自动演化
- 不自动改写核心系统行为
- 第一阶段以技能型经验沉淀为主，不以构建复杂长期记忆系统为目标
- 具体触发与复用策略在实际使用中逐步优化

- Gateway 配置
- 模型配置
- Agent 配置
- Session 相关设置

- 当前模型状态
- 任务执行状态
- 最近错误摘要
- 会话状态
- 基础日志入口
- 工具执行结果可见化
- 后续渠道连接状态

## 15. 成功标准（MVP）

### 15.1 最基础的成功判断

如果第一版达成以下三点，即可视为 MVP 成立：

- 用户能顺利完成安装、配置并启动 LemonClaw
- 用户能在主工作台中完成一次真实任务闭环
- 用户能保存并再次打开会话，继续先前任务上下文，并至少产生一次任务复盘或技能候选记录

### 15.2 主观价值验证

用户应能明确感知到：

- 这不是单纯的 OpenClaw 配置器
- 这不是一次性聊天工具
- 这是一个可以持续使用、持续积累任务经验的桌面 AI Agent 工作台

## 18. 一句话总结

> LemonClaw 第一版的核心，不是做一个功能很多的 Agent 平台，而是先做成一个真正能被持续使用的桌面 AI Agent 工作台：它建立在 OpenClaw 以 Gateway 为核心的统一服务入口、会话路由、Runtime 与本地优先能力之上，通过图形化工作台承接会话治理、多 Agent 与技能沉淀，并为 OpenClaw 的多渠道消息能力保留后续扩展空间，让任务可以被执行、被恢复、被管理、被复用。
